# 报告生成质量优化方案

## 1. 提示词改进与输出格式确认

优先重构提示词，使每一步都有明确输入边界、输出结构和质量验收标准。尤其是章节生成提示词，应要求模型先确认本章的页码范围、核心概念、公式/表格/图示引用，再输出正式讲解内容。

建议把输出格式固定为结构化 Markdown：

- 本章导读：说明本章解决什么问题。
- 页面线索：按页码说明课件推进顺序。
- 核心概念：给定义、直观解释和注意点。
- 公式/表格/图示讲解：必须引用页码，解释变量、结论和使用场景。
- 学习重点与易错点：用于复习。
- 小结：压缩成可记忆的知识框架。

对于 LLM 输出，增加格式确认步骤：生成前在 prompt 中声明必须遵守的 Markdown 层级、页码引用格式、图片引用格式、禁止编造规则；生成后用程序做结构检查，发现缺少页码、标题层级异常、表格损坏、公式未闭合时自动触发修复步骤。

## 2. 更换或分层使用 LLM

报告质量最终高度依赖模型的长文本理解、结构化表达和视觉理解能力。建议采用分层模型策略：

- 结构规划：使用长上下文、稳定遵循 JSON 输出的模型生成目录和章节边界。
- 章节讲解：使用中文表达能力强、长文组织能力好的模型。
- 图表/公式页解释：使用视觉能力强的多模态模型。
- 审校与补漏：使用另一个模型或同模型低温度模式做一致性检查。

如果成本允许，关键步骤不要只依赖单次生成。可采用“生成-审校-局部修复”的链路，通常比单纯换模型更稳定。

## 3. PDF 解析升级为文档元素模型

当前解析主要按页面保存文本、摘要、页面类型和截图。后续应进一步拆成文档元素：

- heading：标题、章节名、页内小标题。
- paragraph：普通正文段落。
- formula：公式区域、LaTeX、变量说明、前后解释文本。
- table：表格区域、Markdown/CSV、表头、单位、注释。
- figure：图示区域、caption、相关正文、截图路径。
- list：项目符号、编号步骤。

每个元素保留页码、bbox、阅读顺序、置信度和来源方式。这样给模型的不是混杂文本，而是带结构的课件内容，能明显减少漏讲和错讲。

## 4. 更可靠的页面重要性分级

页面重要性不应只用于截断文本，更应该决定输入形态：

- 高重要页：保留完整清洗文本、结构化元素和截图。
- 中重要页：保留摘要、关键段落和必要元素。
- 低重要页：保留页面摘要、标题候选和页码角色。
- 公式/表格/图示页：即使文本不长，也应提升优先级，并保留视觉证据。

重要性评分需要结合页面类型、标题候选、关键词、文本密度、公式/表格/图示特征、章节位置和相邻页面关系。不要直接丢弃低分页面，否则会损害报告完整性。

## 5. 公式页识别与提取

公式识别建议采用混合方案：

- 文本规则：检测等号、求和、积分、上下标、希腊字母、矩阵符号、数学关键词。
- 版面规则：识别居中短行、公式块上下留白、公式编号。
- 视觉裁剪：对疑似公式区域裁剪图片。
- 公式 OCR：使用 LaTeX-OCR/Pix2Tex 或多模态模型转 LaTeX。

最终输入给 LLM 时，应提供公式 LaTeX、变量解释、所在页码、前后正文和必要截图。仅给截图会增加模型负担，仅给 OCR 文本又容易丢失结构。

## 6. 表格页识别与提取

表格识别建议优先利用 PDF 版面信息：

- 用 PyMuPDF/pdfplumber 获取文字 bbox、线条、矩形和对齐关系。
- 对有线表格检测横竖线交点和单元格。
- 对无线表格检测列对齐、数字列、短文本块重复。
- 提取失败时再使用 OCR 表格模型或多模态模型。

表格最终应转成 Markdown 表格或 CSV，并附带表格标题、单位、页码、注释和截图。模型生成报告时应被要求解释表格“说明了什么”，而不是简单复述表格内容。

## 7. 最佳 LLM 输入格式

建议采用章节级结构化输入，而不是整份 PDF 直接拼接：

```markdown
# 章节上下文
章节标题：...
页码范围：...
章节目标：...

## 页面 8
页面类型：formula_like
重要性：4.6
标题候选：...
页面摘要：...
清洗正文：...
公式：
1. $$ ... $$
表格：无
图示：page_008.png
提取置信度：text=0.92, formula=0.78
生成要求：解释公式变量、推导逻辑和使用条件。
```

这种输入能让模型清楚知道每页是什么、哪些内容必须讲、哪些内容只是背景，从而提升报告质量并控制 token。

## 8. 质量检查与自动修复

生成后增加程序化检查：

- 是否覆盖每个章节和核心页码。
- 是否引用了公式/表格/图示页。
- Markdown 标题层级是否正确。
- 图片路径是否存在。
- 表格是否能正常渲染。
- 是否出现明显乱码、空章节、重复段落。
- 是否存在“根据课件没有依据”的泛泛表述。

检查失败时，不重新生成整份报告，而是定位到对应章节或段落进行局部修复。

## 9. 人工反馈闭环

前端已有高亮和笔记能力，后续可以把用户标注沉淀为质量数据：

- 用户高亮的段落可视为重要内容。
- 用户笔记可作为下次生成或改写的补充上下文。
- 用户删除/修改频繁的位置可作为质量薄弱点。
- 对同一课程多次生成时，可复用人工修正后的章节结构。

这能让系统从“单次生成”逐渐变成“可持续改进的课程报告生成器”。
